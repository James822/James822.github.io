Best operation syntax for programming languages?

2024/09/26



For the past week or so I've been thinking about ideas for a programming language syntax that I might want to design. I'm not interested in creating a programming language at this point, but I'm wondering if it would be worthwhile to build my own transpiler that compiles down to C code - just because there a couple of things I want to change in C.

For example, consider unary and binary operations. We will ignore the ternary operator, as according to this post: are all n-ary operations are compositions of binary operators?, we can define all n-ary operations with just a composition of binary operations.
In general, what is the best syntax to denote these operations? What I mean specifically is what is the best order to have them? These are the available choices:

    Let x and y be the operands, and * represent the operator, which could be addition "+", subtraction "-", increment "++", or any unary/binary operation.
  • Operators go in-between the operands (Infix Notation): x * y
  • Operators go before the operands (Polish Notation): * x y
  • Operators go after the operands (Reverse Polish Notation): x y *

Personally, I think a mix of Reverse Polish Notation (RPN) and infix is actually the best way to go, for unary and binary operations respectively.
If you use RPN Notation for unary operators (which is basically just putting the operand at the end), and you use infix notation for binary operators; then you can easily define a syntax that has obvious precedent rules AND is very easy to read - IF you take it be the case that unary operators have a higher precedence than binary (which is a standard assumption in most programming languages).

For unary operations it makes intuitive sense when you are reading from left to right. You start with the thing you want to operate on, "x" (which could be any variable or expression), and then you apply the operation "*" to x. So, applying "*" to x is written as:
x*

This scales very well to chained unary operations as well, for example, consider applying the operations "*", "&", and "++" to "x", in that exact order. We would write that as:
x*&++
We don't need brackets or anything, it's just obvious that the operations are applied in that order as you read from left to right.

This notation also scales well to infix notation with binary operations. We can actually consider a binary operation to be a "unary operation" that takes in a paramter for its operation. For example, the "+" binary operation takes two operands, the primary and secondary which are "x" and "y" respectively. In that case, we can think of the binary expression:
x + y
As a unary expression with the form:
x+(y)
Where "y" is seen as the "parameter" to the "unary" operation +. This why it is denoted by "+(y)" because it reads "+ takes y as a parameter".

This left-right consistency between RPN for unary operations and infix-notation for binary operations is why I like this particular mix! It allow us to compose statements such as this:
x*&++ + y++*&* - z&**&
With zero ambiguity, assuming that unary operations have a higher precedence than binary, which is a traditional assumption. The x, y, and z unary "clusters" get evaluated first because unary operators have higher precedence, and then it looks something like this:
x + y - z
Where x + y is evaluated, and then z is subtracted.

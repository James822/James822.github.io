Introduction to Group Theory and Permutations
2025 Mar 07, James Spratt

I've recently taken a course in group theory, branch of abstract algebra that studies the structures known as groups.

Abstract algebra is a fascinating study in mathematics, as it unifies concepts previously scattered in different areas. For example, boolean algbera is the mathematical structure that underlies working with propositional logic, digital circuit design (logic gates), and even set operations. Groups, the primary study of group theory, happen to be structures that are encountered very frequently in mathematics. Many other structures such as vector spaces, rings, fields, the real numbers; can all be shown to have group-like structures.

Group theory is quite a vast topic, so I'm only going to focus on the study of permutation groups in this post. According to Cayley's Theorem, every group is isomoprh. This is another way of saying that permutations can be understood to be the essential concept of group theory, so we can start there.


Permutations

A permutation of a set of numbers {1,2,...,n} is a bijection from {1,2,...,n} to itself. So it is a function of the form:

τ:{1,2,...,n}→{1,2,...,n}

As an example, let us define τ with n=3 to be the following function:

τ:{1,2,3}→{1,2,3}

Given by: τ(1)=2,τ(2)=3,τ(3)=1.

This means simply that 1 is sent to 2, 2 is sent to 3, and 3 is sent to 1.

Rather than painstakingly write out the function value for each input, we can represent this same exact function τ in tabular form as the following:

(123231)

And as you can see, this is the exactly the same format for how we represented the loading zones in the previous table. In general, we can represent any permutation with the following form:

(123...na1a2a3...an)

Where n is the number of elements to be permuted, and ai are the numbers belonging to the set {1,2,...,n}, and it also must be the case that ai≠aj iff i≠j. Any kind of table that follows this specific format is a valid permutation.

It turns out that the set of all permutations for a given set {1,2,...,n} forms a special kind of group called the Symmetric Group.


Symmetric Group

The Symmetric Group for n elements is the set:

Sn={σ:σis a permutation of{1,2,...,n}}

with the operation of function composition given by σ∘τ(a)=σ(τ(a)) for two permutations τ and σ.

For example, if we take two permutations such as these:

τ=(123213),σ=(123231)

The composition of σ and τ is σ∘τ, which means we apply the permutation τ first and then we apply σ second. Composition is read from "right-to-left" which can be confusing.

Let's see where the number "1" goes under the composition of σ∘τ: τ(1)=2, which implies that σ∘τ(1)=σ(τ(1))=σ(2), and therefore: σ(τ(1))=3. A similar process can be performed for "2" and "3" to show that: σ(τ(2))=2, and σ(τ(3))=1.

There is a much easier way to create compositions of permutations by taking it the tabular form as we have been using, and superimpose the first permtuation on top of the second permutation. So take τ and put it on top of σ (order matters here, don't put σ on the top and τ on the bottom), and then you should have this:

(123213) (123231)

Then, rearrange the columns of σ so that the numbers at the top row of σ match the numbers of the bottom row of τ. In this example, all we need to do is swap column 1 and column 2 of σ just like so:

(123213) (213321)

We've changed the arrangment of the columns in σ, but notice that this doesn't change the permutation at all! The permutation stays exactly the same, but we visually changed it just as a helper for our next step.

The next step is to simply remove the bottom row of τ and remove the top row of σ. Basically, remove the middle two rows and leave the very top and very bottom row of the whole thing:

(123321)

But wait, doesn't this just look like another permutation? And in fact it is! You can verify for yourself that we achieved the exact same result as if we evaluated the function for each value. So we've basically proved that the composition of two permutations is also a permutation itself, which is referred to as being closed under the operation of composition.


k-cycles

Now that we have established that the composition of any number of permutations is a single permutation, we can introduce another concept which will help us create a randomizer. Let us define a specific type of permutation called a k-cycle.

A k-cycle is a permutation that takes a collection of distinct numbers in a cycle, leaving all other numbers fixed. For example: 4 -> 2, 2 -> 6, 6 -> 3, 3 -> 4; is a valid cycle because it wraps back around and no number is visited more than once (besides of course the starting number 4, which must be visited twice). The other numbers are left fixed and unchanged.

We can represent this example with the following cycle notation:

(4263)

Each number sends you to the next one, and once you reach the end of the list, it wraps back around to the first element. This example is a 4-cycle as there are 4 distinct numbers in the cycle. But what about the numbers 1 and 5? These numbers are not in the cycle, so they map to themselves: 1 goes to 1 and 5 goes to 5, so we just leave it out of the cycle.

It also doesn't matter where we start the list as long as we maintain the order, so we could also represent the same 4-cycle as so:

(2634)

and you can verify that this changes nothing, but we tend to prefer to start cycles with the least element as a matter of style. Since any k-cycle is also a permutation, we can also represent it using our tabular form like before. The numbers 1 and 5 are not in this cycle which means that they stay fixed as we mentioned before, so we can write:

(2634)=(1234516425)

But can any permutation be represented by a k-cycle? Let's investigate this to see if it is true by considering the following permutation:

(1234521453)

Could we represent this as a single cycle? The answer is no! You can try as hard as you want but there won't be a single k-cycle that can represent this permutation. It turns out however that we can represent this as a composition of 2 cycles, specifically like this:

(1234521453)=(12)(345)

Cycle composition follows the exact same rule as permutation composition like we discussed before, so you first apply the permutation of the first cycle (345), and then you apply the permutation of the second cycle: (12) (recall that permutation composition is right-to-left). The cycles here are also disjoint as they have no elements in common.

It can be proved that every permutation can be represented either as a single cycle, or as a composition of 2 or more disjoint cycles. It is also the case that the order of compositions doesn't matter with disjoint cycles, so the following equality holds for the previous example:

(12)(345)=(345)(12)

The idea behind this is that if a composition of any number of cycles are disjoint, then there are no elements in common, so any element will only be "permuted" by one of the cycles that happens to contain that element. The other cycles will leave that element fixed, so therefore it doesn't really matter where we place the cycle in the composition. It is also possible to use compositions of cycles that are NOT disjoint, but we won't be interested in doing that for our purposes as working with disjoint cycles is much easier.


Converting Between Cycle Form and Tabular Form

There is a straightfoward way to convert from cycle-form to tabular-form so let's take an example composition of disjoint cycles and see how we can convert it to tabular form:

(15)(246)(79)

Since the largest number here is 9, that means that we are dealing with 9 numbers to be permuted, so create an empty table that has numbers 1 through 9 on the top, and leave the bottom row empty:

(123456789.........)

Our goal here is to "fill in the dots" and determine what numbers need to go there. For starters, we observe that the numbers 3 and 8 are not in the cycles whatsoever, so we know that they stay fixed:

(123456789..3....8.)

Then, we start at the number 1 and see where it maps to. Since the only cycle that 1 belongs to is: (15), we know that 1 maps to 5:

(1234567895.3....8.)

Similarly, we can see that 2 maps to 4 because of the cycle (245):

(123456789543....8.)

And if we continue this process we get for the remaining numbers, then we see that we get the following permutation in tabular form:

(123456789543612987)

And you can verify that this is correct.

We also can do the opposite and convert from tabular form to disjoint cycle-form by simply picking numbers and seeing what cycles they belong to. Let's take this same example and convert it back to cycle-form. To start, we pick the number 1 and "follow" it until we get back to 1. In this case, 1 goes to 5, and 5 goes back to 1, so we write that cycle out:

(15)

Then, we pick the next number that has not been visited yet. So we pick 2, and see where that goes. 2 goes to 4, 4 goes to 6, 6 goes to 2; so that should give us (246). Now add this cycle to the existing cycles like so:

(15)(246)

And then we move to the next number that is not already in a cycle, which is 3; and we see that 3 goes to itself, so we can leave it out.

Then the next number that we haven't visited yet is 7. The reason why it is 7 and not 4,5 or 6 is because these numbers are already in the cycles that we have written, so we have already visited them. If we follow 7 then we see that 7 goes to 9 and goes to 7, giving us:

(15)(246)(79)

Finally, the last number we need to visit is 8, but 8 just maps back to itself so we leave it as is.

The fact that every single permutation is a product of disjoint cycles is a very important result because it means we can control the "flow" of how the randomizer works. For example, if the randomized loading zones contained very long cycles, then the player would have to memorize these long cycles in order to navigate the game world effectively. If there are short cycles however, then the player can take advantage of these shorter cycles to help them navigate more easily. As you can see, the compositions of cycles in a paticular randomized game is very important to consider. We might want to control the lengths of these cycles to get certain desired effects.

All of this might seem very trivial to talk about, but consider a game that has 20 loading zones (which is not uncommon, and in fact many games have more), then the amount of permutations of 20 is 20 factorial, which is over 2 quintillion permutations, which is approximately 2 million terabytes of information! The growth of the factorial function is very bad, and it would be absurd to brute force this problem.

Of course, most randomization algorithms will just pick permutations at random, but then they might not have certain properties that we would like. The naive approach would be to generate every permutation and search the permutations to find the desirable ones, but our results from group theory (especially k-cycles) makes it easy to select from literally billions of billions of combinations!

ProtoKyrand Update 1 - Basic Dev Setup

Sept. 29 2024, James Spratt


So, to get started I'll give an overview of my development situation right now.

Currently, I'm hoping that I can do all my development on Linux, and then occasionaly compile on Windows and get it working there. The idea is to strictly use cross-platform libraries and function calls and whatnot so that I should be able to compile on windows (with msvc) with basically no hassle. I doubt this will actually end up working in practice, but we'll deal with the issues as they come up.

I'm also going to do my best to automate and document the dev setup/build process. One thing that I've found over the years is that there are so many little details that can go wrong with the build setup and environment, hat everytime I come back to an old project that's been untouched for some time, it's usually broken in some way and requires several hours and headaches to get working again.

This time, I'll try to get everything right by documenting the build/development setup so that if I need to get someone else involved or remember how to do it myself in the future, it won't be so bad.


Multi-platform workflow

Since I'll be targeting two platforms (Linux and Windows), I'll need to build and test the game on each platform. Currently I'm dual-booting both linux/windows, so I'll be bouncing back and forth compiling and testing. The tentative plan is to develop on linux primarily, and then build/test on windows occasionally.

Since my storage is seperate for both operating systems, I'll need a way to synchronize access between the platforms. For now, I just have an NTFS formatted partition that I'll be using on the windows side. Otherwise, I have a personal drive formatted in ext4 for the main linux development.

I'm not interested in hosting my code on something like github, as I'll be moving large files around and I don't want it hosted on the internet. For now, I'll just keep it local and manually synch the files with two rsync scripts on linux.

All I have to remember is two things:

  • Whenever I want to boot into windows, I'll run an rsync scrip that syncs everything on linux -> to everything on windows
  • Whenever I finish working on Windows, I need to boot back into linux and then run an rsync script to sync everything on windows -> to everything on linux.

I don't think it will be too hard to remember to do this, as long as I make it a habit to always be "aware" that work on windows needs to be synched. The nice thing about this is that when I'm exclusively working on linux, I don't need to think about synchronization; only when I boot into and out of windows do I need to worry about it. In this sense my ext4 drive will be considered the "master" drive.

The documentation for this workflow and the scripts that sync between windows and linux (the NTFS partition and the ext4 partition) will be placed on the NTFS partition that windows has access to. My thinking behind this is that if we didn't have windows, then there would be no need for the syncing workflow and scripts. In that sense, windows is the "entity" that needs to "know" about the synching process.


Future plans for synchronization/collaboration

Currently this multi-platform process is definitely cumbersome and slightly error-prone, but it's also simple and perfectly acceptable at this early stage.

If I start collobarating with other people then that's when things will start to get really interesting. Hosting the code on something like github would be an option, or just using some generic cloud storage solution like AWS, but I'm not so concerned with something like that at this stage.

This feeling that I've described, I want to capture that feeling - in the form of a video game that I want work on that I will call ProtoKyrand.


What in the world does "ProtoKyrand" mean?

For quite some time now I've had ideas for a game that I'd like to create, a game that involves exploration and adventure. I've had a pretty solid idea about it for a couple of years now as I've been keeping a file on it. The file is named "Kyrand", which has no meaning to it other than it was a random fake word that somehow stuck. For some reason I'm drawn to words that start with "K" and that have a "y" in them for whatever reason.

In any case, the name "ProtoKyrand" comes from the fact that I currently don't possess the skills, experience, time, or resources to create this dream game of mine. Instead, I'll work on a prototype of it that includes a reduced set of features but still captures the main idea. That is why I have come to call it "Proto-Kyrand", as it is a prototype of the thing that I eventually want to make.

Kyrand is the "real" game that I eventually want to create, but it's going to take time for that to happen, so in the meantime my plan is to be more realistic about the situation and instead create a very simple "prototype" of the game with a huge reduction in content. The idea is to get this prototype going so that I can garner interest in the project and get some extra help. Hopefully, I'll create something that gains attention by merit of the idea that it demos, rather than the actual product that it is.

Basically, this project is a "test" for many things. It's a test of my skills in game programming and development, a test in storytelling, world design, and just a big experiment in general. I'm excited to see where it leads!


Technical Details

In terms of how I plan to create this game, I basically want to program it from scratch using a basic set of tools.

It may sound overly ambitious that I'm currently planning on creating the game from scratch, but I can't see myself using an engine like Unreal or Unity. I don't think there would be a lot of utility learning how to use these engines because I won't gain the experience of learning how to program a game engine from scratch - which is something that I want to experience firsthand with the hopes that it will make me a better programmer and software engineer. If I were to use an existing engine to create a game (which I have before), my perception is that it won't really develop my skills as a programmer.

So, with that, I'm comitted to creating this game entirely from scratch - With the exception of using some existing libraries and tools. These are the following tools that I'm planning on using, but the list might change later:

  • Programming Language: C

    My programming language of choice will be C, for its simplicity and wide use. C++ is another candidate as well, but its complexity and lack of consistency turns me off, I'd rather just become a really good C programmer and then write tools to accomplish some of the features that C++ has.

  • Graphics API: OpenGL 4

    OpenGL 4 is my choice for the graphics interface as it's well-known and generally simple to use. As a solo "indie" developer, I don't think I really have time to learn another framework like Vulkan or DirectX - I'm mainly just concerned with getting things working at this point, but I'm not opposed to making a switch in the future. OpenGL is also cross-platform which is important as well, because I'm hoping to target Linux and Windows as my main platforms (probably not MacOS or anything else at the moment, I might even drop Linux as a platform if I run into difficulties with it).

  • Platform Library: SDL 2

    SDL is my choice to create the platform layer because of it's widespread use and relatively simple interface. At some point I might consider writing the platform layer myself, but again, at this point, I'm more concerned about just getting some things working first.

For now, these three tools are going to be entirely what I use to create the game and engine, but I'll probably adopt some more tools or change the list as development goes on.
